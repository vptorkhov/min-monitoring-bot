# Папка commands

Эта папка содержит обработчики команд Telegram-бота. Каждая команда реализована в отдельном файле.

## Файлы

### `index.ts`

Центральный файл регистрации всех команд бота. Выполняет:

- Импорт всех обработчиков команд
- Создание необходимых зависимостей (сервисы, репозитории)
- Регистрацию команд в боте

### `start.ts`

Обработчик команды `/start`. Выполняет:

- Проверку существования пользователя
- Запуск процесса регистрации для новых пользователей
- Приветствие зарегистрированных пользователей
- Показ подходящей reply-клавиатуры:
    - `🏠 Выбрать склад`, если склад не выбран
    - `🚲 Взять СИМ` / `🏠 Выбрать склад` / `❌🏠 Отвязаться от склада`, если склад выбран и нет активной сессии
    - `🚲❌ Сдать СИМ`, если склад выбран и активная сессия уже начата

### `set-warehouse.ts`

Обработчик команды `/set_warehouse`. Позволяет:

- Выбрать и прикрепиться к складу
- Показать список активных складов
- Проверить отсутствие активной сессии
- Выбрать склад тремя способами: ручной ввод номера, inline-кнопки `1..N`, reply-кнопки `1..N`
- Отменить шаг выбора склада через `❌ Отмена` (reply keyboard) или `/cancel`
- Запускаться не только из `/set_warehouse`, но и из reply-кнопки `🏠 Выбрать склад` (поддерживается также старый текст `🏠Выбрать склад`)
- После успешного выбора склада показывать основную клавиатуру активного курьера
- Под сообщением об успешном выборе склада показывать inline-клавиатуру действий:
    - `🚲 Взять СИМ`
    - `🏠 Выбрать склад`
    - `❌🏠 Отвязаться от склада`
- При отмене шага выбора склада восстанавливать главную клавиатуру курьера по фактическому состоянию

### `clear-warehouse.ts`

Обработчик команды `/clear_warehouse`. Выполняет:

- Отвязку курьера от всех складов
- Проверку отсутствия активной сессии
- Запуск из `/clear_warehouse` и из reply-кнопки `❌🏠 Отвязаться от склада`
- Запуск из inline-кнопки `❌🏠 Отвязаться от склада` (`callback_query`)
- Восстановление корректной главной клавиатуры после успешной отвязки склада

### `take-sim.ts`

Обработчик команды `/take_sim`. Реализует:

- Показ списка доступных СИМ на складе
- Показ inline-кнопок `1..N` под сообщением со списком СИМ
- Показ reply-клавиатуры выбора после списка: `❌ Отмена`, затем кнопки `1..N`
- Создание новой сессии использования
- После успешного начала сессии показ кнопки `🚲❌ Сдать СИМ`
- Приоритет личных устройств
- Запуск из `/take_sim` и из reply-кнопки `🚲 Взять СИМ`
- Запуск из inline-кнопки `🚲 Взять СИМ` (`callback_query`)
- Обработку inline-кнопок `1..N` как выбора соответствующего номера из списка
- Обработку `❌ Отмена` как эквивалента `/cancel` в процессе выбора СИМ

### `return-sim.ts`

Обработчик команды `/return_sim`. Выполняет:

- Завершение активной сессии
- Запуск из `/return_sim` и из reply-кнопки `🚲❌ Сдать СИМ`
- На этапе вопроса `Есть ли повреждение у СИМ?` показывает inline-кнопки `Нет` / `Да`
- Нажатие inline-кнопок отправляет в чат соответствующий ответ (`Нет` или `Да`) и продолжает сценарий
- На этапе `Выберите тип повреждения` показывает:
    - reply-клавиатуру `Слабое` / `Критическое` / `❌ Отмена`
    - inline-кнопки `Слабое` / `Критическое` под сообщением
- Нажатие inline-кнопок типа повреждения отправляет в чат соответствующий ответ (`Слабое` или `Критическое`) и продолжает сценарий
- Опрос о повреждениях для не-личных устройств
- Обновление статуса устройства
- Возврат основной клавиатуры активного курьера после успешной сдачи СИМ (если снова выполнены условия показа)

### `cancel.ts`

Обработчик команды `/cancel`. Предоставляет:

- Сброс текущего состояния пользователя
- Очистку временных данных
- Сообщение об отмене действия
- Восстановление главной клавиатуры по текущему состоянию курьера (склад выбран/не выбран, есть ли активная сессия)

### `admin.ts`

Обработчики скрытого админского режима:

- `/admin` — принудительно останавливает текущий курьерский сценарий (регистрация, выбор склада, взятие/сдача СИМ и т.д.) и переводит пользователя в админский режим
- `/admin_login` — пошаговый вход администратора (логин -> пароль -> авторизованный админский режим)
- `/admin_register` — реальный пошаговый сценарий регистрации админа (логин -> пароль -> создание неактивной записи)
- `/admin_change_password` — смена пароля для авторизованного админа/суперадмина: бот запрашивает новый пароль (не менее 6 символов), при валидном вводе обновляет пароль в БД и оставляет пользователя в авторизованном режиме
- `/admin_set_warehouse` — выбор склада для админа любого уровня доступа: бот выводит сообщение «Выберите номер склада» и нумерованный список активных складов, ожидает номер; при некорректном вводе показывает ошибку и продолжает ожидание; после успеха сохраняет `admins.warehouse_id` и переводит в состояние админа с выбранным складом
- `/admin_clear_warehouse` — отвязка админа от склада; доступна только при выбранном складе; очищает `admins.warehouse_id`, сообщает об успехе, показывает обновленное сообщение о доступных командах и переводит в состояние админа без выбранного склада
- `/admin_sim_interactions` — сценарий взаимодействия с СИМ выбранного склада; показывает нумерованный список СИМ (без личного транспорта), принимает выбор по номеру или номеру СИМ, затем выводит карточку СИМ и доступные подкоманды
- `/admin_sim_change_active` — доступна только после выбора СИМ через `/admin_sim_interactions`; изменяет статус активности (`is_active`) по вариантам `1`/`2` или текстовым значениям (`Активный`/`Отключен`, регистронезависимо); недоступна при активной сессии по выбранному СИМ
- `/admin_sim_change_status` — доступна только после выбора СИМ через `/admin_sim_interactions`; изменяет статус исправности (`status`) по вариантам `Исправен`/`Поврежден`/`Сломан`; недоступна при активной сессии; при выборе `Сломан` дополнительно устанавливает `is_active=false`
- `/admin_sim_story` — доступна только после выбора СИМ через `/admin_sim_interactions`; показывает историю всех сессий выбранного СИМ (время ЧЧ:ММ по Москве, ФИО курьера, состояние после сессии, комментарий) и автоматически возвращает к выбору команды для этого СИМ
- `/admin_sim_delete` — доступна только после выбора СИМ через `/admin_sim_interactions`; удаляет СИМ после подтверждения `ДА`; если по СИМ есть активная сессия, удаление запрещено; если есть только завершенные сессии, СИМ удаляется вместе с историей в `session`
- `/superadmin_create_warehouse` — создание нового склада суперадмином; доступно только авторизованному админу с `permissions_level >= 2` (обычному админу отправляется сообщение «Нет прав на эту команду»); двухшаговый сценарий: название (мин. 2 символа) → адрес (мин. 2 символа) → запись в БД; доступна независимо от того, выбран ли склад; во время сценария отменяется по `/cancel` (c возвратом в авторизованный режим) или `/exit_admin`
- `/superadmin_edit_warehouses` — запуск сценария редактирования складов суперадмином (`permissions_level >= 2`): бот выводит нумерованный список складов (включая отключенные) и ждёт номер склада; при невалидном номере выводится ошибка и повторно ожидается ввод
- `/superadmin_edit_warehouse_name` — доступна только внутри активного сценария `/superadmin_edit_warehouses`; бот просит новое название (мин. 2 символа), обновляет запись и возвращает к выбору действия по выбранному складу
- `/superadmin_edit_warehouse_address` — доступна только внутри активного сценария `/superadmin_edit_warehouses`; бот просит новый адрес (мин. 2 символа), обновляет запись и возвращает к выбору действия по выбранному складу
- `/superadmin_edit_warehouse_status` — доступна только внутри активного сценария `/superadmin_edit_warehouses`; поддерживает ввод: `1`, `1.`, `Активный`, `1. Активный`, `2`, `2.`, `Отключен`, `2. Отключен` (регистронезависимо для слов), обновляет `is_active`; при попытке отключения склада с активными сессиями возвращает ошибку
- `/superadmin_edit_warehouse_delete` — доступна только внутри активного сценария `/superadmin_edit_warehouses`; бот запрашивает подтверждение `ДА` (строго в верхнем регистре), после успешного удаления возвращает пользователя в состояние до запуска базовой команды
- `/superadmin_edit_admins` — запуск сценария редактирования администраторов суперадмином (`permissions_level >= 2`): бот выводит нумерованный список администраторов (активных и неактивных, кроме суперадминов) и ждёт номер администратора; при невалидном номере выводится ошибка и повторно ожидается ввод
- `/superadmin_edit_admin_status` — доступна только внутри активного сценария `/superadmin_edit_admins`; бот показывает текущий статус и ждёт ввод: `1`, `1.`, `Активный`, `1. Активный`, `2`, `2.`, `Отключен`, `2. Отключен` (регистронезависимо для слов), затем обновляет `is_active` и возвращает к выбору действия по администратору
- `/superadmin_edit_admin_delete` — доступна только внутри активного сценария `/superadmin_edit_admins`; бот запрашивает подтверждение `ДА` (строго в верхнем регистре), после успешного удаления показывает обновленный список администраторов и возвращает к выбору номера
- `/superadmin_edit_admin_password` — доступна только внутри активного сценария `/superadmin_edit_admins`; бот запрашивает новый пароль (мин. 6 символов), обновляет пароль администратора и возвращает к выбору действия
- `/admin_apply_registrations` — доступна авторизованному админу и суперадмину (как с выбранным складом, так и без него); бот показывает нумерованный список неактивных курьеров без записей в `session` в формате `N. ФИО nickname` (если `nickname` отсутствует — только `ФИО`), после выбора номера запрашивает подтверждение `Да`/`Нет`; при `Да` активирует курьера (`is_active=true`) и возвращает к обновленному списку, при `Нет` оставляет курьера неактивным и также возвращает к обновленному списку
- `/admin_edit_couriers` — доступна авторизованному админу и суперадмину только при выбранном складе; выводит нумерованный список курьеров выбранного склада, ожидает номер и затем показывает карточку курьера (ФИО, телефон, никнейм, склад, статус, активная сессия при наличии)
- `/admin_edit_courier_status` — доступна только после выбора курьера через `/admin_edit_couriers`; меняет `is_active` по вводу `1`/`1.`/`Активный`/`1. Активный` или `2`/`2.`/`Отключен`/`2. Отключен` (текст — регистронезависимо), после успеха возвращает к обновленной карточке курьера
- `/admin_courier_history` — доступна только после выбора курьера через `/admin_edit_couriers`; показывает последние 50 сессий курьера (новейшие → старейшие) в формате `номер СИМ начало:ДД.ММ.ГГГГ ЧЧ:ММ конец:ДД.ММ.ГГГГ ЧЧ:ММ` и ожидает `ДА` (строго верхний регистр) для полной истории или `/cancel`/`нет` для возврата
- `/superadmin_edit_couriers` — доступна только суперадмину (`permissions_level >= 2`), запускается как при выбранном складе, так и без него; показывает список всех курьеров, включая курьеров без выбранного склада
- `/superadmin_edit_courier_status` — аналог `/admin_edit_courier_status`, но внутри сценария `/superadmin_edit_couriers`
- `/superadmin_courier_history` — аналог `/admin_courier_history`, но внутри сценария `/superadmin_edit_couriers`
- `/admin_logout` — выход из авторизованного админ-режима в предадминское состояние
- `/exit_admin` — полный выход из всех админских состояний в курьерский режим
- Пока админ не авторизован, в админском режиме разрешены только: `/admin`, `/admin_login`, `/admin_register`, `/admin_logout`, `/exit_admin`, `/cancel`
- Все остальные курьерские команды в этом режиме блокируются guard-проверкой
- В сценарии редактирования складов:
    - `/cancel` на этапе выбора склада/действия возвращает в состояние до `/superadmin_edit_warehouses`
    - `/cancel` на этапах ввода нового названия/адреса/статуса/подтверждения удаления возвращает к выбору действия по выбранному складу
    - ввод подкоманд `/superadmin_edit_warehouse_*` без активного контекста выбора склада приводит к сообщению о недоступности команды
- В сценарии редактирования администраторов:
    - `/cancel` на этапе выбора администратора возвращает в состояние до `/superadmin_edit_admins`
    - `/cancel` на этапе выбора действия возвращает к выбору администратора с повторным выводом списка
    - `/cancel` на этапах ввода статуса/подтверждения удаления/нового пароля возвращает к выбору действия по выбранному администратору
    - ввод подкоманд `/superadmin_edit_admin_*` без активного контекста выбора администратора приводит к сообщению о недоступности команды
- В сценарии смены пароля:
    - `/cancel` на этапе ввода нового пароля возвращает пользователя в `admin_authenticated`
- В сценарии `/admin_set_warehouse`:
    - `/cancel` на этапе ожидания номера склада возвращает пользователя в состояние, из которого была запущена команда
- В сценарии `/admin_sim_interactions`:
    - на этапе списка СИМ бот явно подсказывает `/cancel - вернуться в состояние выбранного склада`
    - `/cancel` на этапе выбора СИМ возвращает в `admin_authenticated_with_warehouse`
    - `/cancel` на этапах выбора действия/изменения/подтверждения удаления возвращает к обновленному списку СИМ
    - на этапе выбора действия по конкретному СИМ бот явно подсказывает `/cancel - вернуться к списку СИМ`
    - если список СИМ пуст, выполняется возврат в `admin_authenticated_with_warehouse`
- В сценарии `/admin_apply_registrations`:
    - `/cancel` на этапе выбора номера курьера возвращает в состояние, из которого была вызвана команда
    - `/cancel` на этапе подтверждения `Да`/`Нет` возвращает к обновленному списку неактивных курьеров
    - если при таком возврате список кандидатов пуст, выполняется возврат в состояние до запуска команды
- В сценариях `/admin_edit_couriers` и `/superadmin_edit_couriers`:
    - `/cancel` на этапе выбора номера курьера возвращает в состояние до запуска базовой команды
    - `/cancel` на этапе выбора действия по курьеру возвращает к списку курьеров
    - `/cancel` на этапах ввода статуса или ожидания полной истории возвращает к карточке выбранного курьера
    - на этапе ожидания полной истории ответ `нет` (в любом регистре) возвращает к карточке выбранного курьера

## Архитектура команд

Каждая команда:

1. Проверяет условия выполнения (регистрация, активная сессия и т.д.)
2. Выполняет основную логику
3. Отправляет ответ пользователю
4. Может устанавливать состояния для многошаговых процессов

Inline-callback обработка организована через единый роутер `src/bot/callback-router.ts`:

- В командах регистрируются callback-обработчики, вместо отдельных `bot.on('callback_query')`
- Это позволяет централизованно завершать callback через `answerCallbackQuery`
- Логика callback в командах стала предсказуемее и проще для расширения
